# 解码与采样策略

> 模型每一步输出的是一个词表上的概率分布，「如何从分布里选出下一个 token」就是解码策略。它直接决定生成结果的确定性、多样性与质量，是高频但容易被忽视的考点。

## 从 logits 到 token

模型最后一层输出每个 token 的 **logits**，经过 softmax 得到概率分布，再由解码策略决定取哪个 token。temperature、top-k、top-p 等参数都作用在这一步。

## 确定性解码

### 贪心解码（Greedy）

每步直接取概率最大的 token。简单、快、可复现，但容易陷入重复、平庸，且「局部最优 ≠ 全局最优」。

### 束搜索（Beam Search）

每步保留概率最高的 B 条候选序列（beam），最终选整体概率最高的。

- 优点：比贪心更可能找到高概率序列，适合**翻译、摘要**等有「标准答案」的任务。
- 缺点：开销大；在开放式生成中反而导致**枯燥、重复、安全平庸**的输出，所以对话/创作场景很少用。

## 随机采样

### 温度（Temperature）

在 softmax 前把 logits 除以温度 T：

$$p_i = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)}$$

- **T → 0**：分布变尖锐，趋近贪心，更确定、更保守。
- **T = 1**：保持原始分布。
- **T > 1**：分布变平缓，更随机、更有创造性，但也更容易胡说。

### Top-k 采样

只在概率最高的 k 个 token 里按概率重新归一化后采样。缺点是 k 固定：分布很尖时 k 太大会引入噪声，分布很平时 k 太小又限制多样性。

### Top-p / 核采样（Nucleus Sampling）

从「累积概率达到 p 的最小 token 集合」里采样（如 p=0.9）。**候选集大小随分布动态变化**——确定时候选少、不确定时候选多，比 top-k 更自适应，是目前最常用的采样方式。

### 常配合的惩罚项

- **重复惩罚（Repetition Penalty）/ frequency_penalty / presence_penalty**：降低已出现 token 的概率，缓解复读。
- **min-p**：以「最高概率的某个比例」为阈值过滤，鲁棒性更好。

## 进阶解码

- **对比解码（Contrastive Decoding）**：用大模型与小模型的 logits 之差选词，抑制小模型也会犯的低级错误。
- **投机解码（Speculative Decoding）**：小模型起草、大模型验证，是**加速**手段而非改变分布，详见 [推理优化与部署](/inference/inference-optimization)。
- **约束解码（Constrained / Guided Decoding）**：强制输出符合 JSON Schema、正则、语法，常用于结构化输出与 Function Calling。

## 参数怎么调（实战速记）

| 场景 | 建议 |
| --- | --- |
| 事实问答 / 代码 / 抽取 | 低温度（0～0.3），甚至贪心，追求确定可复现 |
| 翻译 / 摘要 | Beam Search 或低温度 |
| 创意写作 / 头脑风暴 | 较高温度（0.7～1.0）+ top-p 0.9 |
| 需要结构化输出 | 低温度 + 约束解码 |

## 高频追问

**Q：temperature 和 top-p 能一起用吗？** 能，且常一起用。一般先用 temperature 调整分布形状，再用 top-p 截断长尾。但通常**不建议两个都设成很激进的值**，否则难以预测；多数情况下固定一个、调另一个。

**Q：为什么开放式生成不用 Beam Search？** Beam Search 偏好高概率序列，而高概率往往意味着「常见、安全、重复」，导致输出枯燥；开放式生成需要多样性，采样类方法更合适。

**Q：温度为 0 一定完全可复现吗？** 理论上趋近贪心，但实际受 GPU 浮点非确定性、批处理顺序、并行归约等影响，仍可能有微小差异；要严格复现还需固定随机种子和后端设置。

**Q：top-k 和 top-p 的本质区别？** top-k 固定候选**数量**，top-p 固定候选**累积概率**。top-p 能根据分布的确定程度自适应调整候选集大小，更鲁棒，因此更主流。

**Q：生成总是重复怎么办？** 加重复惩罚 / no-repeat n-gram、适当提高温度或 top-p、改用采样而非贪心、检查是否训练数据或模板有问题。
