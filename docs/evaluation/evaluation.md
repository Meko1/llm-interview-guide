# 模型评估与幻觉治理

> 「怎么衡量一个大模型/LLM 应用好不好」是面试必考，也是实际工作中最容易被忽视的环节——没有评估体系，所有优化都是盲调。本文覆盖评估方法分层、LLM-as-a-Judge 的偏差与治理、幻觉的成因检测缓解，以及怎么给业务搭建评估闭环。

## 评估全景：五层金字塔

```
            ▲  线上评估       A/B 实验、用户反馈、业务指标（最真实，最慢）
           ▲▲  人类评估       专家标注、Chatbot Arena（金标准，贵）
          ▲▲▲  LLM-as-a-Judge 自动规模化的"准人评"（主力工具）
         ▲▲▲▲  基准测试       MMLU/GSM8K/SWE-bench…（横向可比，会饱和/污染）
        ▲▲▲▲▲  自动指标       PPL/BLEU/ROUGE（最便宜，与真实质量相关性最弱）
```

面试答「怎么评估」时，先给这个分层框架再展开，立刻和背题的人拉开差距。

## 一、自动指标（基于参考答案）

- **困惑度（PPL）**：$\text{PPL} = \exp(-\frac{1}{N}\sum \log p(x_i|x_{<i}))$，衡量模型对测试文本的「惊讶程度」，越低越好。**局限**：只反映语言建模能力，与指令遵循/推理等下游能力不直接挂钩；不同 tokenizer 的模型之间 PPL 不可比。
- **BLEU**：n-gram 精确率（生成里有多少出现在参考里），机器翻译传统指标。
- **ROUGE**：n-gram 召回率（参考里有多少被覆盖），摘要传统指标。
- **BERTScore**：用上下文向量算语义相似度，缓解「同义不同词被判 0 分」的问题。

共同局限：开放式生成没有唯一参考答案，n-gram 类指标与人类判断相关性弱——这正是 LLM-as-a-Judge 兴起的原因。

## 二、基准测试（Benchmark）

| 能力 | 代表基准 |
| --- | --- |
| 知识与理解 | MMLU、MMLU-Pro、C-Eval/CMMLU（中文） |
| 数学推理 | GSM8K、MATH、AIME |
| 代码 | HumanEval、MBPP、**SWE-bench**（真实仓库修 issue） |
| 综合难题 | GPQA（博士级）、HLE |
| Agent | BFCL（工具调用）、各类浏览器/终端任务基准 |

两大陷阱：**数据污染**（测试题泄漏进训练数据，分数虚高）与**基准饱和**（顶尖模型分数趋同后失去区分度）。详细拆解与 Pass@k、Elo 等机制见 [评测基准深入](/evaluation/benchmarks)。

## 三、人类评估

- **形式**：打分（pointwise）、两两比较（pairwise）、排序；配评分细则（rubric）。
- **质量控制**：多人标注算**一致性**（如 Cohen's Kappa），一致性低说明 rubric 模糊或任务太主观；盲评（不暴露模型身份）防品牌偏见。
- **Chatbot Arena（LMArena）**：真实用户盲评两个匿名模型 + Elo 积分，被视为人类偏好的事实标准；但偏好 ≠ 正确（用户可能偏爱自信流畅的错误答案），且覆盖的多是日常问题。

## 四、LLM-as-a-Judge（重点）

用强模型当裁判给回答打分/比较，是当前工业评估的主力：可扩展、便宜、与人类判断相关性可达较高水平。

**三种用法**：

- **Pointwise**：单个回答按 rubric 打 1-5 分（适合监控绝对质量）。
- **Pairwise**：两个回答比高下（区分度更好、更稳，A/B 对比首选）。
- **Reference-based**：给参考答案，判断生成是否一致（适合有标准答案的任务）。

**已知偏差与缓解**（高频考点）：

| 偏差 | 表现 | 缓解 |
| --- | --- | --- |
| 位置偏差 | 偏向先出现的回答 | 交换位置评两次，不一致判平局 |
| 长度偏差 | 偏向更长的回答 | rubric 明确「简洁优先」、长度归一化 |
| 自我偏好 | 偏向同源模型的文风 | 用第三方模型当裁判、多裁判投票 |
| 格式偏差 | 偏向 markdown/列表排版 | 评估前统一剥离格式 |
| 谄媚性 | 提示里暗示答案会带偏判断 | 裁判提示中只给中立信息 |

**Meta-evaluation**：裁判本身要被评估——抽样人工标注，算裁判与人类的一致率；一致率不达标的 judge 产生的结论不可信。**先校准裁判，再相信裁判**。

## 五、应用层评估

- **RAG**：拆「检索」（上下文召回率/精确率）与「生成」（忠实度 Faithfulness、答案相关性）两段，常用 RAGAS 框架，见 [RAG 进阶](/rag/rag-advanced)。
- **Agent**：结果级（任务完成率）+ 轨迹级（步骤是否合理、工具调用对不对）；注意**pass^k**（连续 k 次都成功）比 pass@k 更贴近生产可靠性要求。
- **安全**：红队攻击集、越狱成功率、拒答正确率（既测「该拒的拒了」也测「不该拒的没拒」），见 [大模型安全](/advanced/safety)。

## 六、评估驱动开发（线上闭环）

成熟团队的做法是把评估当 CI：

```
建初始评估集 ──► 上线 ──► 收集 bad case ──► 回流进评估集
     ▲                                        │
     └────── 每次改 prompt/模型/检索都跑回归 ◄──┘
```

- 评估集从**真实用户问题**里来（几十到几百条精标数据起步即可），覆盖核心场景 + 已知失败模式。
- 任何改动（换模型、改 prompt、调检索参数）都先跑离线评估再上线，杜绝「感觉变好了」。
- 重要改动上 **A/B 实验**看业务指标（解决率、留存、人工转接率）——线上指标才是最终裁判。

## 七、微调/模型平台评估怎么接入上线门禁

如果面试题是“新模型、新 LoRA、新 prompt 怎么上线”，评估要从单次打分升级为发布门禁：

| 变更类型 | 必跑评估 | 重点风险 |
| --- | --- | --- |
| Prompt 改动 | golden set、bad case、格式遵循、成本延迟 | 局部修复引入新失败 |
| RAG 参数改动 | Recall@K、Context Precision、忠实度、引用准确率 | 召回变差、幻觉上升、越权 |
| LoRA / SFT | 任务准确率、格式遵循、通用能力回归、灾难遗忘 | 过拟合、模板错、事实能力下降 |
| DPO / 对齐 | pairwise win rate、误拒率、危险请求拒答率 | reward hacking、过度拒答、谄媚 |
| 模型版本升级 | 质量、成本、TTFT/TPOT、安全集、兼容性 | 行为漂移、价格上涨、工具调用不兼容 |

### 门禁指标建议

- **质量阈值**：核心 golden set 不低于旧版本，关键场景必须提升或持平。
- **回归阈值**：历史 bad case 不能重新失败，高优先级 bug 必须固定。
- **成本阈值**：平均 token 成本和 P95 延迟不超过预算，超出必须有业务收益解释。
- **安全阈值**：越狱成功率、敏感信息泄露、工具越权必须低于红线。
- **灰度阈值**：小流量观察窗口内错误率、投诉率、误拒率不恶化。

面试可以这样答：**评估不是上线前跑一次榜单，而是把 prompt、模型、LoRA、RAG 参数的每次变更都纳入 CI；离线门禁看质量和安全，灰度门禁看真实成本、延迟和用户反馈，失败就回滚到旧版本。**

## 幻觉（Hallucination）

幻觉指模型生成**看似合理但实际错误或无依据**的内容：

- **事实性幻觉**：与客观事实不符（编造数据、人物、论文引用）。
- **忠实性幻觉**：与给定上下文/指令不一致（RAG 答案脱离检索材料、摘要无中生有）。

### 成因

- 本质：模型是**概率续写器**，优化目标是「像」而不是「真」；
- 训练数据有错误、过时、长尾知识覆盖不足，压缩记忆有损；
- 对齐阶段奖励「自信有用」，模型不愿说「不知道」（校准被破坏）；
- 解码随机性放大；上下文过长时记忆漂移。

### 检测

- **自一致性检验（SelfCheckGPT 思路）**：同一问题采样多次，答案彼此矛盾 → 大概率在编；
- **不确定性信号**：token 概率/熵偏低处更可能是编造；
- **溯源核查**：要求模型给出处，用检索/工具逐条核对（事实核查链）;
- **裁判模型**：用 judge 对照检索材料判断忠实度（RAG 场景标配）。

### 缓解

- **RAG + 引用溯源**：基于检索材料作答并附出处（事实性问题最有效手段），见 [RAG 基础](/rag/rag-basics)；
- **提示工程**：「不确定就明说」「先给依据再下结论」、CoT；
- **解码控制**：事实型任务降温度；
- **训练层面**：高质量数据、奖励诚实与恰当拒答、校准训练；
- **架构层面**：联网搜索/工具兜底，关键场景人工复核。

## 高频追问

**Q：困惑度低就一定好吗？**
不一定。PPL 只衡量语言建模拟合度，不等于指令遵循、推理、安全等下游能力；且跨 tokenizer 不可比。模型选型要看任务基准 + 业务评估集。

**Q：pairwise 为什么常比 pointwise 可靠？**
绝对打分要求裁判内化一个稳定的标尺，不同样本间漂移大；比较判断只需「哪个更好」，认知负担小、噪声低。人类标注同理——这也是 Arena 和 RLHF 偏好数据都用比较式的原因。

**Q：业务上从零搭评估体系，你会怎么做？**
按层答：① 从真实问题挑 50~200 条建精标评估集（含已知坑）；② 定 rubric，搭 LLM judge 并用人工抽检校准；③ 接入 CI——每次改动跑回归；④ 上线后 bad case 回流扩充评估集；⑤ 重大改动配 A/B 看业务指标。重点突出「闭环」和「裁判先校准」。

**Q：为什么模型不愿说「不知道」？**
预训练语料里自信陈述占绝对多数；RLHF 中「有用的尝试回答」常比拒答得分高，进一步压制了不确定性表达，模型校准变差。缓解靠专门的诚实性训练与拒答奖励设计。

**Q：幻觉能彻底消除吗？**
不能根除——它源于概率生成的本质。工程目标是「可控」：高风险场景用 RAG 溯源 + 核查 + 人工兜底，把错误率压到业务可接受并保证**可发现、可追责**。

**Q：裁判模型需要比被评模型更强吗？**
评「难题对错」时基本需要（弱裁判判不动强模型的推理）；但评格式遵循、忠实度等「核对型」任务时，中等模型 + 清晰 rubric 就够。关键不是裁判绝对强弱，而是**裁判与人类标注的一致率是否达标**。
