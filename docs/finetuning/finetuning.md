# 微调范式（SFT / PEFT）

> 微调让通用基座模型适配具体任务、风格或领域。本文系统梳理全参微调、SFT、参数高效微调（PEFT）的全貌、数据构造、灾难性遗忘与选型。LoRA 细节见 [LoRA / QLoRA 详解](/finetuning/lora)，对齐见 [RLHF / DPO](/finetuning/rlhf)。

## 一、微调在大模型生命周期中的位置

```
预训练(学知识) ──▶ SFT(学听指令) ──▶ 对齐 RLHF/DPO(学人类偏好)
   Base 模型          Instruct/Chat 模型        对齐模型
```

[预训练](/pretraining/pretrain) 产出只会续写的 Base 模型；**微调（尤其 SFT）是把它变成会对话、会遵循指令的 Chat 模型的关键一步**。

## 二、微调的种类

| 类型 | 说明 |
| --- | --- |
| **全参数微调（Full FT）** | 更新模型所有参数，效果好但显存/算力开销巨大 |
| **参数高效微调（PEFT）** | 只训练少量新增/选定参数，冻结大部分权重，省显存 |
| **指令微调（SFT）** | 用「指令-回答」对训练，让基座变成会对话/遵循指令的模型（是一种训练目标，可用全参或 PEFT 实现） |

> 注意：SFT 是「**训练什么**」（用指令数据、监督学回答），PEFT 是「**怎么训**」（只动少量参数）。二者正交——SFT 可以全参做，也可以用 LoRA 做。

## 三、SFT（监督微调）深入

SFT 做法：构造大量高质量的 `(指令, 期望回答)` 样本，用标准的下一个 token 预测损失训练，但通常**只对「回答」部分计算 loss**（指令/prompt 部分做 mask）。

### 关键点

- **数据质量 > 数量**：少量（几千到几万条）高质量、多样化的指令数据，往往胜过海量低质数据——这就是 LIMA 的「Less Is More for Alignment」结论。
- **对话模板（Chat Template）**：必须用与模型一致的角色标记格式（system/user/assistant 及特殊 token），否则推理时会错乱。这是新手常踩的坑。
- **数据多样性**：覆盖不同任务类型、难度、长度、领域，避免能力偏科。
- **数据构造方法**：人工标注、从强模型蒸馏（用 GPT-4 生成）、Self-Instruct（用模型自举生成指令）、真实日志清洗。

### SFT 数据示例（带模板与 loss mask）

```
<|system|>你是一个有帮助的助手
<|user|>把这句话翻译成英文：今天天气真好        ← mask，不算 loss
<|assistant|>The weather is really nice today.   ← 只对这部分算 loss
```

## 四、PEFT 常见方法

| 方法 | 原理 | 特点 |
| --- | --- | --- |
| **LoRA / QLoRA** | 给权重加低秩旁路，只训小矩阵 | 最主流，零推理延迟，详见 [LoRA](/finetuning/lora) |
| **Adapter** | 层间插入小瓶颈网络 | 增加推理延迟（串行新模块） |
| **Prefix-Tuning** | 在每层注意力前拼可学习的前缀向量 | 只训前缀 |
| **P-Tuning v2** | 类似 Prefix，深层加软提示 | 适合理解类任务 |
| **Prompt-Tuning** | 仅在输入嵌入层加少量可训练 token | 最轻量，规模大时才有效 |
| **BitFit** | 只训练偏置项（bias） | 极致省参数 |
| **(IA)³** | 学习缩放向量调制激活 | 参数极少 |

> PEFT 的共同思路：**冻结绝大部分预训练权重，只训练极少量参数**，从而省显存、防遗忘、便于多任务管理。

## 五、灾难性遗忘（Catastrophic Forgetting）

微调特定任务后，模型可能丢失原有的通用能力。缓解方法：

- **混入通用数据（replay）**：微调数据里掺一部分通用/原始能力数据一起训练。
- **用 PEFT**：如 LoRA 只动少量参数，基座能力基本保留。
- **控制学习率和轮数**：避免过拟合到狭窄分布。
- **较小的可训练参数量**：动得越少，遗忘越少。

## 六、微调 vs RAG vs Prompt（选型）

| 需求 | 首选 |
| --- | --- |
| 改变风格、格式、语气、特定任务能力 | **微调** |
| 注入实时/私有事实知识、需溯源、频繁更新 | **RAG**（见 [RAG](/rag/rag-basics)） |
| 简单任务、快速验证、通用能力 | **Prompt**（见 [Prompt 工程](/prompt/prompt-engineering)） |

三者常组合：微调改表达 + RAG 供知识 + Prompt 调用。

## 七、面试专项：SFT/PEFT 数据与训练排障

面试官问“微调后效果不好怎么办”，不要只说调学习率。按 **数据 -> 模板 -> 训练 -> 评估 -> 上线** 五层定位：

| 现象 | 常见原因 | 排查与修复 |
| --- | --- | --- |
| 推理时角色错乱、重复 system prompt | 训练和推理 chat template 不一致 | 固定 tokenizer 的官方 template；训练样本保留 role 标记；上线前用相同模板回放验证 |
| 训练 loss 降了，业务效果没升 | 数据低质、蒸馏答案啰嗦或错误 | 抽样人工审查；删除无信息样本；提高困难样本和真实 bad case 占比 |
| 验证集先升后降 | 学习率过大、epoch 过多、LoRA 容量过大 | 降 lr、减少 epoch、增加 dropout、降低 r 或扩大验证集 |
| 通用问答能力下降 | 灾难性遗忘 | 混入通用能力 replay 数据；使用 LoRA/QLoRA；保留通用基准回归 |
| 线上 JSON/表格格式漂移 | 训练样本格式不统一，或只靠 prompt 约束 | 统一输出 schema；把格式错误样本加入 SFT；上线继续用结构化解析和重试兜底 |
| 领域事实仍然答错 | 把微调当知识库使用 | 新知识、权限和引用交给 RAG；微调只做表达、格式和任务习惯 |
| 无法证明微调更好 | 没有独立评估集和 baseline | 建 golden set、bad case set、通用能力集；对比 base、prompt-only、RAG、LoRA 版本 |

**面试答法：**
我会先确认数据和模板，因为这两个问题最常见；再看学习率、epoch、LoRA 容量是否导致过拟合；最后用独立评估集拆业务指标、通用能力、格式遵循和安全拒答。微调不是为了替代 RAG，事实知识仍然要用检索和引用保证可控。

## 八、高频追问

**Q：SFT 和 PEFT 是一回事吗？** 不是。SFT 是训练目标（用指令-回答数据监督学习「如何回答」）；PEFT 是训练方式（只更新少量参数）。SFT 可以全参实现，也可以用 LoRA 等 PEFT 实现，二者正交。

**Q：SFT 阶段为什么只对回答算 loss？** 我们希望模型学会「如何回答」，而非拟合用户的提问；对指令算 loss 可能让模型学会复述问题，反而有害。所以 prompt 部分做 mask、只对 assistant 回答算损失。

**Q：为什么数据质量比数量重要（LIMA）？** 基座在预训练已学到大部分知识，SFT 主要是「激发并对齐到指令格式」，少量高质量、多样的样本就足以做到；海量低质数据反而引入噪声、降低效果。

**Q：对话模板为什么重要？** 模型在训练时按特定角色标记格式学习，推理时必须用一致的模板（system/user/assistant 及特殊 token），否则模型「认不出」对话结构，输出错乱。这是常见的踩坑点。

**Q：什么时候该微调，什么时候用 RAG/Prompt？** 改风格/格式/能力 → 微调；注入实时/私有事实、需更新和溯源 → RAG；简单任务快速验证 → Prompt。三者常组合。

**Q：全参微调和 LoRA 怎么选？** 数据多、追求极致效果、资源充足 → 全参；资源有限、需快速迭代或维护多任务版本 → LoRA/QLoRA。多数业务 LoRA 性价比最高。

**Q：微调后效果变差可能是什么原因？** 数据质量差/有噪声、对话模板不一致、学习率过大、过拟合、灾难性遗忘、样本分布与评测分布不匹配、训练轮数过多。

**Q：怎么缓解灾难性遗忘？** 混入通用数据 replay、用 PEFT（LoRA）只动少量参数、控制学习率和训练轮数、减少可训练参数量。

**Q：SFT 数据怎么来？** 人工标注、从更强模型蒸馏（GPT-4 生成）、Self-Instruct（模型自举生成指令-回答）、真实业务日志清洗。关键是质量、多样性和与目标场景的匹配。
