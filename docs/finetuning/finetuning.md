# 微调范式（SFT / PEFT）

> 微调让通用基座模型适配具体任务或风格。本文梳理全参微调、SFT 与参数高效微调（PEFT）的全貌。

## 微调的种类

| 类型 | 说明 |
| --- | --- |
| 全参数微调（Full Fine-Tuning） | 更新模型所有参数，效果好但显存/算力开销巨大 |
| 参数高效微调（PEFT） | 只训练少量新增/选定参数，冻结大部分权重，省显存 |
| 指令微调（SFT） | 用「指令-回答」对训练，让基座变成会对话/遵循指令的模型 |

## SFT（Supervised Fine-Tuning，监督微调）

SFT 是把 Base 模型变成 Chat/Instruct 模型的关键一步。做法：构造大量高质量的 `(指令, 期望回答)` 样本，用标准的下一个 token 预测损失训练，但通常**只对「回答」部分计算 loss**（指令部分做 mask）。

关键点：

- **数据质量 > 数量**：少量（几千到几万条）高质量、多样化的指令数据，往往胜过海量低质数据（LIMA 的「Less Is More」结论）。
- **对话模板（Chat Template）**：必须用与模型一致的角色标记格式（system/user/assistant），否则推理时会错乱。
- **数据多样性**：覆盖不同任务类型、难度、长度，避免能力偏科。

## PEFT 常见方法

- **LoRA / QLoRA**：给权重矩阵加低秩旁路，只训练小矩阵。最主流，详见 [LoRA / QLoRA 详解](/finetuning/lora)。
- **Adapter**：在层间插入小型瓶颈网络，只训练它。会增加推理延迟。
- **Prefix-Tuning / P-Tuning**：在输入前拼接可学习的「软提示」向量，只训练这些前缀。
- **Prompt-Tuning**：更轻量的软提示，只在输入嵌入层加少量可训练 token。
- **BitFit**：只训练偏置项（bias），极致省参数。

## 灾难性遗忘（Catastrophic Forgetting）

微调特定任务后，模型可能丢失原有的通用能力。缓解方法：

- 混入部分通用数据（replay）一起训练；
- 用 PEFT（如 LoRA）只动少量参数，保留基座能力；
- 控制学习率和训练轮数，避免过拟合；
- 采用较小的可训练参数量。

## 高频追问

**Q：什么时候该微调，什么时候该用 RAG/Prompt？**
- 改变**风格、格式、语气、特定任务能力** → 微调；
- 注入**实时/私有的事实知识**、且知识频繁更新 → RAG（微调难以可靠注入并更新事实，还易引发幻觉）；
- 简单任务、快速验证 → 先试 Prompt 工程。
三者常组合使用。

**Q：全参微调和 LoRA 怎么选？** 数据多、追求极致效果且资源充足 → 全参；资源有限、需要快速迭代或维护多个任务版本 → LoRA。

**Q：SFT 阶段为什么只对回答算 loss？** 我们希望模型学会「如何回答」，而不是去拟合用户的提问；对指令算 loss 可能让模型学会复述问题，反而有害。

**Q：微调后效果变差可能是什么原因？** 数据质量差/有噪声、对话模板不一致、学习率过大、过拟合、灾难性遗忘、样本分布与评测分布不匹配等。
