# Function Calling 与 MCP

> 工具调用是 Agent 的核心能力。本文讲清 Function Calling 的机制，以及新兴的 MCP 协议。

## Function Calling（函数调用）

Function Calling 是让模型「调用外部函数/工具」的能力。注意：**模型本身不执行函数**，它只负责输出「我想调用哪个函数、用什么参数」这一结构化意图，真正的执行由外部程序完成。

典型流程：

1. **声明工具**：开发者把可用函数以 schema（名称、描述、参数 JSON Schema）形式告诉模型。
2. **模型决策**：模型根据用户请求判断是否需要调工具，需要则输出结构化的调用请求（函数名 + 参数）。
3. **外部执行**：你的程序解析该请求，真正执行函数（如查数据库、调 API），拿到结果。
4. **结果回传**：把执行结果作为新消息喂回模型。
5. **生成回答**：模型综合结果生成最终自然语言回答（或继续发起下一次调用）。

模型之所以会用工具，是因为在训练（尤其对齐阶段）中专门学习了根据工具描述生成合法调用的能力。**工具的 description 写得好不好，直接决定模型用得准不准。**

## 为什么需要 Function Calling？

- 突破知识截止：调用搜索/实时 API 获取最新信息；
- 精确计算：交给计算器/代码执行，避免模型算错；
- 连接真实世界：操作数据库、发邮件、下单等；
- 是构建 Agent 和 RAG（检索作为工具）的基础。

## MCP（Model Context Protocol）

MCP 是 Anthropic 于 2024 年提出的**开放标准协议**，用于规范「LLM 应用 ↔ 外部数据源/工具」之间的连接，被形象地称为「AI 应用的 USB-C 接口」。

**要解决的问题**：在 MCP 之前，每个应用接入每个工具/数据源都要写一套定制集成，是 M×N 的组合爆炸。MCP 提供统一标准，让工具方只需实现一个 MCP Server，应用方只需做一个 MCP Client，变成 M+N。

**核心架构（C/S）**：

- **MCP Host / Client**：AI 应用一侧（如 Claude Desktop、IDE），发起连接和调用。
- **MCP Server**：暴露能力的一侧，提供三类资源：
  - **Tools（工具）**：可被模型调用的函数（如查询数据库）。
  - **Resources（资源）**：可读取的数据/文件（如文档内容）。
  - **Prompts（提示模板）**：预定义的提示词模板。

**与 Function Calling 的关系**：Function Calling 是「模型如何表达想调用工具」的能力；MCP 是「工具如何被标准化地提供和发现」的协议。MCP 让工具可复用、可组合、生态化，Function Calling 是其底层能力之一。

## 高频追问

**Q：模型会真的执行函数吗？** 不会。模型只输出调用意图（函数名+参数），执行由外部代码完成，结果再回传给模型。这点常被问，务必答清楚。

**Q：Function Calling 失败/参数错误怎么办？** 校验参数（JSON Schema 校验）、把错误信息回传让模型重试、限制工具数量与复杂度、提供清晰的工具描述和示例。

**Q：MCP 解决了什么痛点？** 把工具集成从 M×N 定制降为 M+N 标准化，实现工具的复用与生态共享，降低 Agent 接入成本。

**Q：工具太多模型选错怎么办？** 控制单次可见工具数量、用检索动态筛选相关工具（tool RAG）、清晰命名与描述、分层/分组管理工具。
