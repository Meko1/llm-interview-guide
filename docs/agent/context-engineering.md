# 上下文工程（Context Engineering）

> 2025 年，业界共识从「Prompt 工程」升级到「上下文工程」——决定 Agent 成败的，往往不是某句提示词，而是**喂进上下文窗口的全部信息如何组织**。这是当下最热的工程话题之一。

## 从 Prompt 工程到上下文工程

- **Prompt 工程**：聚焦「怎么写好一句/一段指令」，主要面向单轮、静态场景。
- **上下文工程**：聚焦「在每一步推理时，如何动态地把**正确的信息、以正确的格式、在正确的时机**放进上下文窗口」。

> 一句话：Prompt 工程是「写好提示词」，上下文工程是「管好整个上下文窗口」。Agent 时代，后者更关键——因为 Agent 是多步、有状态、要调工具的，上下文在不断变化。

## 上下文里都有什么？

一个 Agent 的上下文窗口通常包含：

| 成分 | 说明 |
| --- | --- |
| System Prompt | 角色、目标、规则、可用工具说明 |
| 对话历史 | 多轮交互记录 |
| 检索内容（RAG） | 从知识库召回的相关资料 |
| 工具定义与结果 | 工具 schema + 历次调用的返回 |
| 记忆 | 长期记忆检索回来的信息（见 [Agent 记忆系统](/agent/agent-memory)） |
| 当前任务状态 | 计划、中间结果、待办 |

上下文工程要解决的核心矛盾：**上下文窗口有限且越长越贵越慢、还有「lost in the middle」**，但 Agent 需要的信息却越来越多。

## 核心技术

### 1. 写入（Write）：把信息留在窗口外

- **Scratchpad（草稿区）**：把中间思考/计划写到上下文外的暂存区，需要时再取回。
- **记忆**：把重要信息存入长期记忆（向量库/数据库），见 [记忆系统](/agent/agent-memory)。

### 2. 选择（Select）：只放该放的

- **RAG / 按需检索**：用到再查，而不是全塞进去。
- **工具按需加载**：工具多时用检索筛出当前相关的少数工具（tool RAG），避免模型选错、也省 token。
- **示例按需挑选**：动态选最相关的 few-shot 例子。

### 3. 压缩（Compress）：把长的变短

- **摘要（Summarization）**：把冗长的历史/工具结果压成摘要。
- **裁剪（Trimming）**：按规则丢弃旧的、不相关的内容。

### 4. 隔离（Isolate）：拆分上下文

- **多 Agent 分治**：把任务拆给多个 Agent，每个只维护自己的小上下文，避免单一上下文臃肿（见 [多 Agent](/agent/multi-agent)）。
- **子上下文/沙箱**：让某些处理在独立上下文里进行，只把结论带回主线。

## 上下文退化的常见失败模式

长上下文、多轮 Agent 容易出现的「上下文病」：

- **Context Poisoning（污染）**：错误信息（如一次幻觉）进入上下文后被反复引用、滚雪球。
- **Context Distraction（分心）**：无关内容太多，淹没了关键信息。
- **Context Confusion（混淆）**：堆了太多工具/信息，模型选错、用错。
- **Context Clash（冲突）**：上下文里存在互相矛盾的信息。

> 这些正是「上下文不是越多越好」的具体表现，呼应 [长上下文专题](/basics/long-context) 的 lost-in-the-middle。

## 高频追问

**Q：上下文工程和 Prompt 工程有什么区别？** Prompt 工程关注「写好单条提示词」；上下文工程关注「在多步、有状态的 Agent 运行中，动态管理整个上下文窗口里放什么、怎么放、何时放」。Agent 时代上下文工程更核心。

**Q：上下文工程的核心手段有哪些？** 概括为四类：写入（scratchpad/记忆，把信息存到窗口外）、选择（RAG/工具按需加载，只放相关的）、压缩（摘要/裁剪）、隔离（多 Agent/子上下文分治）。

**Q：为什么上下文不是越多越好？** 受窗口上限、成本、延迟约束，且存在 lost-in-the-middle 和上下文退化（污染、分心、混淆、冲突）。把无关信息塞进去反而降低准确率，关键是「精准」而非「堆量」。

**Q：工具很多时怎么管理上下文？** 不要把所有工具 schema 都塞进去，用检索按当前任务动态筛出相关工具（tool RAG），清晰命名分组，控制单次可见工具数，减少混淆和 token 消耗。

**Q：什么是 context poisoning？怎么防？** 错误信息（常来自一次幻觉或坏的工具结果）进入上下文后被后续步骤反复引用、不断放大。防范：加校验/反思节点、对工具结果做核查、必要时清理或重置上下文、用引用溯源。
