# RAG 基础与流程

> RAG 是当下大模型应用最热门的方向之一。本文讲清它解决什么问题、完整流程与核心组件。

## 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）= **检索外部知识 + 让模型基于检索结果生成回答**。简单说：先从知识库里找到相关资料，再把资料连同问题一起喂给模型，让它「开卷答题」。

它解决了大模型的几个痛点：

- **知识陈旧**：模型知识截止于训练时间，RAG 可接入实时/最新数据。
- **缺乏私有知识**：模型不知道企业内部文档，RAG 可外挂私有知识库。
- **幻觉**：让模型基于给定材料作答并附出处，显著降低编造。
- **可溯源**：能给出引用来源，便于核查。

相比微调，RAG 更新知识只需更新知识库（无需重训），成本低、可控性强。

## 完整流程

RAG 分为**离线索引（Indexing）**和**在线检索生成（Retrieval + Generation）**两大阶段。

### 离线：构建索引

1. **加载（Load）**：读取各种来源文档（PDF、网页、数据库等）。
2. **切分（Chunk）**：把长文档切成小块（chunk）。切分策略很关键：块太大检索不精准、太小丢失上下文。常用按固定长度 + 重叠（overlap）、按语义/标题结构切分。
3. **向量化（Embed）**：用 Embedding 模型把每个 chunk 编码成向量。
4. **入库（Store）**：把向量和原文存入向量数据库（如 Faiss、Milvus、Qdrant、pgvector）。

### 在线：检索与生成

5. **查询向量化**：把用户问题用同一 Embedding 模型编码。
6. **检索（Retrieve）**：在向量库中做相似度检索（如余弦相似度），召回 Top-K 个最相关 chunk。
7. **（可选）重排（Rerank）**：用更精细的 Rerank 模型对召回结果重新排序，提升精度。
8. **增强（Augment）**：把检索到的内容填进 prompt 模板，和问题一起组成上下文。
9. **生成（Generate）**：LLM 基于上下文生成带依据的回答。

## 核心组件

- **Embedding 模型**：把文本映射为语义向量，决定召回质量。常用 BGE、M3E、OpenAI text-embedding 等。
- **向量数据库**：存储和高效检索向量，支持 ANN（近似最近邻，如 HNSW、IVF）。
- **Rerank 模型**：交叉编码器（cross-encoder），把 query 和文档拼一起精细打分，比向量召回更准但更慢，故只用于对召回的少量结果重排。

## 向量检索原理

Embedding 把语义相近的文本映射到向量空间中相近的位置。检索时计算 query 向量与文档向量的相似度（余弦相似度/内积），取最近的若干个。由于精确检索在海量向量上太慢，实际用 **ANN（近似最近邻）** 算法（HNSW、IVF-PQ 等）在精度和速度间折中。

## 高频追问

**Q：RAG vs 微调，怎么选？** 知识频繁更新、需要溯源、私有事实多 → RAG；改变风格/格式/固定能力 → 微调。两者可结合：微调改善表达，RAG 提供知识。

**Q：为什么需要 chunk overlap（重叠）？** 防止把一个完整语义单元从中间切断，重叠能保证关键信息不因切分边界而丢失。

**Q：召回不准怎么办？** 见 [RAG 进阶与优化](/rag/rag-advanced)：查询改写、混合检索、重排、调整切分策略、换更强的 Embedding 模型等。

**Q：为什么向量召回 + Rerank 两段式？** 向量召回（bi-encoder）快、可在海量库上粗筛；Rerank（cross-encoder）准但慢，只对粗筛出的几十条精排，兼顾效率与精度。
