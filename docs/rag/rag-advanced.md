# RAG 进阶与优化

> 朴素 RAG 在真实场景常常召回不准、答非所问。本文梳理优化手段与评估方法，是进阶面试的高频区。

## 朴素 RAG 的常见问题

- 召回内容不相关或缺失关键信息；
- query 与文档表述不一致导致召回失败（语义鸿沟）；
- 切分不当导致上下文断裂；
- 检索到太多无关内容稀释了有效信息；
- 多跳问题（需要综合多个文档）回答不好。

优化可从**检索前（Pre-retrieval）、检索中（Retrieval）、检索后（Post-retrieval）**三个环节切入。

## 检索前优化：查询改写

- **Query Rewriting / 扩展**：用 LLM 改写、澄清、扩展用户问题，弥补语义鸿沟。
- **HyDE（假设性文档嵌入）**：先让 LLM 生成一个「假想答案」，再用它去检索——因为答案和文档在语义空间更接近。
- **Multi-Query**：让 LLM 把一个问题拆成多个子查询，分别检索后合并，提升召回覆盖。
- **Step-back Prompting**：先退一步问更抽象的问题，召回背景知识。

## 检索中优化：混合检索

- **混合检索（Hybrid Search）**：结合**稠密检索（向量，擅长语义）**和**稀疏检索（BM25/关键词，擅长精确匹配专有名词、术语）**，用 RRF（倒数排名融合）等方式融合结果，互补提升召回。
- **元数据过滤**：结合时间、来源、标签等结构化过滤，缩小检索范围。
- **更优的切分**：父子分块（小块检索、大块喂给模型）、按语义/标题层级切分。

## 检索后优化

- **Rerank（重排）**：用 cross-encoder 对召回结果精排，只保留最相关的少数 chunk。性价比很高的一步。
- **上下文压缩 / 过滤**：去掉检索结果中的无关句子，只保留与问题相关的内容，节省上下文并减少干扰。
- **重排序位置**：把最相关的内容放在 prompt 首尾（缓解 lost in the middle）。

## 高级 RAG 范式

- **多路召回 + 融合**：多种检索器并行召回再融合。
- **Self-RAG / 自反思**：模型自行判断是否需要检索、检索结果是否有用、答案是否有依据。
- **GraphRAG**：构建知识图谱，适合需要跨文档关联、全局总结的复杂问题。
- **Agentic RAG**：把检索当作 Agent 可调用的工具，模型自主决定何时检索、检索什么、是否多轮检索。

## RAG 评估

分两部分评估：

- **检索质量**：命中率（Hit Rate）、MRR、召回率、上下文相关性（Context Relevance）。
- **生成质量**：
  - **忠实度（Faithfulness）**：回答是否忠于检索到的内容（不编造）。
  - **答案相关性（Answer Relevance）**：回答是否切题。
  - **上下文精确率/召回率**：检索内容是否覆盖且不冗余。

常用框架：**RAGAS**、TruLens 等，多采用 LLM-as-a-Judge 自动评测。

## 高频追问

**Q：召回率高但回答还是差，问题在哪？** 可能是生成环节：上下文过长稀释信息、相关内容位置不佳、prompt 模板不好，或检索内容虽相关但不足以回答（需要多跳/补充检索）。

**Q：chunk 大小怎么定？** 没有银弹，需实验。常见 256～512 token + 10～20% 重叠。可用父子分块兼顾检索精度与上下文完整。

**Q：Embedding 模型怎么选？** 看任务领域、语言（中文选 BGE/M3E 等）、向量维度与成本，参考 MTEB 榜单，并在自己的数据上评测。领域差异大时可微调 Embedding。
