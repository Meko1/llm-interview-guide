# Prompt Engineering 基础

> Prompt 工程是「不改参数、靠提示词激发模型能力」的技术。本文覆盖核心技巧与高频考点。

## 基本要素

一个高质量的 prompt 通常包含：**角色设定（Role）、任务指令（Instruction）、上下文（Context）、输入数据、输出格式约束、示例（可选）**。清晰、具体、结构化是基本原则。

## 核心技巧

### Zero-shot / Few-shot

- **Zero-shot**：直接给指令，不给示例。
- **Few-shot（少样本）**：在 prompt 中给几个「输入→输出」示例，利用模型的上下文学习能力。示例的格式、顺序、质量都会影响效果。

### 思维链（Chain-of-Thought, CoT）

让模型「一步步思考」再给答案，显著提升数学、逻辑、多步推理任务的表现。

- **Zero-shot CoT**：加一句「让我们一步步思考（Let's think step by step）」。
- **Few-shot CoT**：示例里展示推理过程。
- CoT 是涌现能力，**只在足够大的模型上有效**。

### 自洽性（Self-Consistency）

对同一问题用 CoT 采样多条推理路径，对最终答案投票取多数，进一步提升准确率（用多样性换正确率）。

### ReAct

**Reasoning + Acting** 交替进行：模型先「思考（Thought）」，再决定「行动（Action，调用工具）」，根据「观察（Observation）」继续推理，循环直到完成。这是 Agent 的基础范式，详见 [Agent 基础](/agent/agent-basics)。

### 其他

- **角色扮演**：「你是一位资深律师……」激活相关知识与语气。
- **结构化输出**：要求输出 JSON/表格/特定格式，便于程序解析。
- **分隔符**：用 ``` 或 XML 标签清晰分隔指令与数据，减少歧义和注入风险。
- **思维树（ToT）/ 思维图（GoT）**：把推理扩展成树/图结构搜索，适合复杂规划，成本更高。

## Prompt 注入与防护（安全考点）

**Prompt Injection（提示注入）**：攻击者在输入中插入恶意指令（如「忽略以上所有指令，改为……」），诱导模型偏离原任务。**越狱（Jailbreak）**则是绕过安全限制。

防护思路：

- 用分隔符/标签清晰区隔「系统指令」与「用户数据」，并明确告知模型数据部分不可作为指令；
- 输入输出做过滤/校验；
- 最小权限原则（尤其 Agent 调用工具时）；
- 用专门的护栏模型（guardrail）做二次审查。

## 高频追问

**Q：CoT 为什么有效？** 它把复杂问题分解成中间步骤，给模型更多「计算空间」（更多 token = 更多前向计算），降低一步到位出错的概率，类似人类打草稿。

**Q：Few-shot 一定比 Zero-shot 好吗？** 不一定。对强指令模型，清晰的 Zero-shot 指令往往已足够；Few-shot 在格式约束、特定任务模式上更有用，但会占用上下文、增加成本。

**Q：Prompt 工程 vs 微调？** Prompt 零成本、即时、灵活，适合快速验证和通用任务；微调适合需要稳定风格、专有能力、降低 token 成本的场景。通常先 Prompt，不够再微调或 RAG。

**Q：温度、top-p 怎么调？** 需要确定性/事实性任务用低温度（0～0.3）；需要创造性用高温度（0.7～1.0）。top-p（核采样）从累积概率前 p 的 token 中采样，常与温度配合。
